# EMA Trend Filter для сигналов памп/дамп

## Проблема

Бот шлёт алерты о памп/дамп-движениях (6%+ за окно), но не даёт понять, стоит ли входить в сделку по этому сигналу. Нужен фильтр на основе тренда и положения цены относительно EMA — чтобы видеть сразу в алерте, совпадает ли сигнал со старшим трендом и не убежала ли цена слишком далеко от точки разумного входа.

Автотрейдинг не нужен — только более информативный сигнал для ручного входа.

## Референс

Логика подсмотрена в открытом проекте [Lermont/HTX-crypto-bot](https://github.com/Lermont/HTX-crypto-bot) (EMA Pullback Bot): старший таймфрейм определяет направление тренда по EMA, младший — ищет откат цены к EMA и подтверждение возобновления движения.

## Архитектура

Новая функция `get_trend_verdict(symbol: str) -> dict` вызывается в `process_kline()` (`crypto_bot.py:660-726`) сразу после того как `triggered=True` (после строки 690), до построения графика и caption. Использует существующий `get_klines(symbol, interval, limit)` (`crypto_bot.py:176`) — два дополнительных REST-запроса к Binance (1h и 15m), только в момент сработавшего сигнала, не на каждый тик вебсокета.

## Логика тренда и пуллбека

- **1H (тренд):** EMA50 и EMA200 на последних ~210 свечах.
  - Тренд вверх: EMA50 > EMA200
  - Тренд вниз: EMA50 < EMA200
- **15m (положение относительно EMA):** EMA20 на последних ~30 свечах.
  - Расстояние: `abs(price - EMA20) / EMA20 * 100`
  - Порог "у EMA": `EMA_DISTANCE_THRESHOLD_PCT` (по умолчанию 0.5%), хранится в `bot_settings.json` рядом с существующими параметрами (`MIN_VOLUME_USDT`, `PUMP_PCT`, `IIV_HOT`, `MIN_PAIR_AGE_DAYS`)

## Вердикт (4 состояния)

| Вердикт | Условие |
|---|---|
| 🟢 По тренду, у EMA | направление сигнала (памп→лонг, дамп→шорт) совпадает с 1H-трендом И расстояние до EMA20 ≤ порога |
| 🟡 По тренду, далеко от EMA | тренд совпадает, но расстояние > порога |
| 🔴 Против тренда | направление сигнала противоречит 1H EMA50/EMA200 |
| ⚪ Не удалось проверить | ошибка/таймаут REST-запроса klines |

## Формат сообщения

`fmt_caption()` (`crypto_bot.py:29`) получает новый параметр `trend_verdict: str`. Пример строки в подписи:

```
Тренд: 🟢 По тренду (EMA50>EMA200) | Цена у EMA20 (+0.3%)
```

Алерт отправляется всегда, вердикт не блокирует и не задерживает отправку.

## Обработка ошибок

Если запрос klines для 1h/15m падает (таймаут, ошибка Binance) — вердикт `⚪ Не удалось проверить`, сигнал всё равно уходит в Telegram без задержки. Основной поток сигналов (пуш в вебсокете) не блокируется дополнительными REST-запросами — они выполняются точечно, только при сработавшем триггере.

## Тестирование

- Юнит-тест на фикстурных сериях свечей: аптренд (EMA50>EMA200 + цена у EMA20), даунтренд, боковик/против тренда — проверяем что `get_trend_verdict` возвращает ожидаемый вердикт.
- Тест на ошибку REST-запроса (мок `get_klines` бросает исключение) → вердикт `⚪`.
- Ручная проверка на сервере: сравнить 2-3 реальных сигнала с TradingView вручную после деплоя.

## Вне рамок

- Автотрейдинг (открытие/закрытие позиций) — не делаем, только информационный вердикт.
- Скоринг с весами, относительная сила к BTC, объёмный спайк на пуллбеке — возможное развитие, не в этой итерации.
